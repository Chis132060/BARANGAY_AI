# Database Schema

## Purpose

This document defines the intended baseline schema for Smart Barangay.

## Overview

The schema is organized around identity, resident records, service workflows, communications, AI knowledge, and governance. This document is the contract to convert into SQLAlchemy models, Alembic migrations, and Supabase policies.

## Architecture

| Table | Purpose | Key Relationships |
| --- | --- | --- |
| `user_profiles` | Application profile for Supabase Auth users | `auth_user_id`, role links |
| `resident_profiles` | Resident demographic and contact data | `user_profile_id`, `household_id` |
| `staff_profiles` | Staff employment and assignment data | `user_profile_id` |
| `roles` | Named system roles | permissions through `role_permissions` |
| `permissions` | Fine-grained permission codes | roles through `role_permissions` |
| `households` | Household-level grouping | residents and addresses |
| `addresses` | Structured location data | households and residents |
| `service_categories` | Service grouping | service types |
| `service_types` | Available barangay services | categories and requests |
| `service_requests` | Resident service submissions | resident, service type, assigned staff |
| `request_status_history` | Status transition history | request and actor |
| `request_attachments` | Uploaded request documents | request and storage object |
| `announcements` | Public or targeted notices | author profile |
| `notifications` | Notification delivery records | recipient profile |
| `device_tokens` | Push notification tokens | user profile |
| `knowledge_documents` | Approved source documents for RAG | chunks |
| `knowledge_chunks` | Searchable text chunks | document and embeddings |
| `knowledge_embeddings` | Vector embeddings | chunk |
| `conversation_sessions` | AI chat sessions | owner profile |
| `conversation_messages` | AI/user messages | session |
| `audit_logs` | Security and workflow audit trail | actor and target metadata |

## Implementation Details

Recommended common columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `uuid` | Primary key generated by database |
| `created_at` | `timestamptz` | Default `now()` |
| `updated_at` | `timestamptz` | Updated by trigger or application |
| `deleted_at` | `timestamptz` | Nullable soft-delete marker where needed |
| `created_by` | `uuid` | References `user_profiles.id` for auditable records |

Important constraints:

| Entity | Constraint |
| --- | --- |
| `service_requests.status` | Must be one of `draft`, `submitted`, `under_review`, `approved`, `rejected`, `completed`, `cancelled` |
| `request_status_history` | Must preserve previous and next status |
| `knowledge_documents.status` | Must be one of `draft`, `approved`, `archived` |
| `device_tokens.token` | Unique per platform/user combination |
| `audit_logs` | Append-only by application convention and restricted policies |

## Design Decisions

The schema uses normalized tables for core government records and keeps metadata JSON only for extensibility points such as audit target details and service-specific form fields. Service-specific fields may be stored as validated JSONB initially, then promoted to relational columns when reporting or validation needs justify it.

## Advantages

- Gives a stable foundation for migrations and API contracts.
- Keeps service workflows auditable.
- Allows AI knowledge ingestion without mixing unapproved documents into retrieval.

## Disadvantages

- Some service-specific data may require later normalization.
- UUID indexes are larger than integer indexes.
- Strict constraints require careful migration planning.

## Security Considerations

Resident tables, attachments, conversations, and audit logs require RLS and backend authorization. PII columns should be minimized, masked in logs, and excluded from analytics exports unless explicitly permitted.

## Performance Considerations

Add btree indexes on ownership and status columns. Add HNSW or IVFFlat indexes for embeddings after enough rows exist to tune search quality. Use composite indexes for staff queues such as `(assigned_staff_id, status, created_at)`.

## Future Improvements

- Convert this contract into Alembic migrations.
- Add generated ERD artifacts from the live schema.
- Add seed data for service categories and roles.
- Add retention and archival tables after policy approval.
